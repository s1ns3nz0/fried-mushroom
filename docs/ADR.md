# Architecture Decision Records

## 철학
MBCRA 방법론의 3단계 구조(핵심기능 식별 → 위협경로 이해 → 위험 우선순위화)를 UAV 전 위협 유형으로 확장. "최적"은 생존이 아니라 임무 성공 기여도로 정의한다. AI는 판단 재료를 다듬는 역할까지만 하고, 최종 결정 로직(RAC, 매트릭스, 하드코딩 임계값)은 결정론적으로 유지해 추적 가능성을 확보한다 (MIL-STD-882E SCC-1 원칙).

---

### ADR-001: Python 순수함수 파이프라인
**결정**: 각 레이어를 `src/onboard/layer_XX_*/run.py`의 `run(input, ...) -> dict` 순수 함수로 구현. 클래스/DI 프레임워크 없음.
**이유**: D4D 문서의 각 레이어가 "입력 스키마 → 처리 로직 → 출력 스키마"로 이미 순수함수 형태로 정의돼 있어 그대로 매핑 가능. TDD 하기 쉬움. 사이클마다 상태를 재구성해도 성능 문제 없음(온보드 컴퓨트 실 배포 시 최적화 여지 남김).
**트레이드오프**: 사이클 간 상태 전달(quality_delta의 이전 quality 등)이 호출자 책임이라 파이프라인 오케스트레이터가 조금 복잡해짐.

### ADR-002: AI 스텁 우선, 실제 모델은 후순위
**결정**: 03 Sensor Abstraction의 AI 채널(proximity_object=YOLO/MobileNet, terrain_class 카메라 보조, acoustic YAMNet 2차)은 MVP에서 stub 함수로 고정값 반환. 채널 인터페이스만 정의.
**이유**: 파이프라인 구조·데이터 흐름 검증이 최우선. 실제 모델 로딩은 SWaP 튜닝, ONNX 변환, 데이터셋 라이선스 등 별도 축이라 MVP 스코프를 흐린다.
**트레이드오프**: 골든 케이스로 통합 검증 가능하지만 실 위협 시나리오에서의 정확도는 이 MVP로 검증 불가. 04~07 결정론 로직만 실전 검증됨.

### ADR-003: 결정론 로직과 AI 강화판을 물리적으로 분리
**결정**: 각 레이어에서 결정론적 계산 결과와 AI 강화판 결과를 별도 함수/필드로 산출. 05의 `RAC_MATRIX` 조회는 AI가 절대 오버라이드 불가. 04의 `confidence`, 05의 `continuous_L/S`, 06의 `ai_reliability`는 병렬 참고지표.
**이유**: MIL-STD-882E SCC-1(Autonomous Software Control)이 요구하는 "추적 가능한 결정 로직" 원칙. AI가 매트릭스를 바꾸면 인증 불가.
**트레이드오프**: 같은 정보를 두 번 계산(결정론적/AI). CPU 오버헤드 증가하지만 04~05가 이미 GPU 미사용이라 실질 부담 낮음.

### ADR-004: JSON dict I/O, 스키마는 TypedDict로 선언
**결정**: 레이어 간 통신은 JSON-직렬화 가능한 dict. 타입은 `src/onboard/shared/schemas.py`에 TypedDict로. 런타임 검증(pydantic 등)은 후순위.
**이유**: D4D 문서가 이미 JSON 스키마로 인터페이스를 정의. 로깅·리플레이·디버깅이 자연스러움. 개발 초기엔 타입 힌트로 IDE 검증만으로 충분.
**트레이드오프**: 런타임 스키마 위반이 즉시 잡히지 않음. 통합 테스트로 커버.

### ADR-005: 5개 마스터 시나리오 골든 케이스
**결정**: T1(GPS 스푸핑), T2(사이버), T3(근접 소화기), T4(물리 포획), T7(지형충돌) 각각에 대해 mock 원시 센서→종단 응답까지 골든 JSON을 `examples/`에 두고 통합 테스트로 회귀 방지.
**이유**: D4D 문서의 예시 JSON이 이미 하나(T3) 있고, 나머지 위협도 문서에 예상 결과가 서술돼 있어 재사용 가능. E2E 회귀가 곧 스펙 회귀.
**트레이드오프**: 문서 파라미터를 바꾸면 골든을 함께 업데이트해야 함. 이건 오히려 강제하고 싶은 규율.
