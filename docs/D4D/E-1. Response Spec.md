# E-1. Response Spec

`06. Response`의 기술 세부 문서입니다. 실행 가능한 코드, 손계산 검증, 파라미터 출처를 담습니다. 개념·근거·쉬운 설명은 `06. Response`를 참고하고, 이 문서는 "정확히 어떻게 계산되는가"에 집중합니다.

입력은 `05. Risk Assessment`의 `run_risk_assessment()` 출력(`candidates[]`, priority_rank로 정렬됨, `ambient_rac`)입니다.

---

## 1. flight_action / comms_level — RAC=High 테이블

```python
from risk_assessment_v2 import THREAT_CATEGORY

FLIGHT_ACTION_HIGH_EVASIVE = {
    "PHYSICAL": "RTL",
    "REMOTE": "REROUTE",
    "NAVIGATION": "ALTITUDE_CHANGE_REROUTE",
}
COMMS_LEVEL_HIGH_EVASIVE = {
    "PHYSICAL": "L3", "REMOTE": "L2", "NAVIGATION": "L2",
}
FLIGHT_ACTION_HIGH_EARLY = "POSTURE_ELEVATE"
COMMS_LEVEL_HIGH_EARLY = "L1"
# POSTURE_ELEVATE는 이미 RAC=High(임계 근접)인데 아직 kill_chain_stage=초기 —
# 예방적 고도상승(07의 POSTURE_ELEVATE_ALTITUDE_M=+25m) + 05 재평가 주기 단축으로
# 킬체인 진행을 더 빨리 잡아냄. (신규 확정: 이전엔 special_action=None인 미정 상태였음)
SPECIAL_ACTION_HIGH_EARLY = "INCREASE_ASSESSMENT_FREQUENCY"
```

## 2. flight_action / comms_level / special_action — RAC=Serious/Medium/Low

```python
FLIGHT_ACTION_BY_RAC = {"Serious": "ALTITUDE_CHANGE", "Medium": "MAINTAIN", "Low": "MAINTAIN"}
COMMS_LEVEL_BY_RAC = {"Serious": "L1", "Medium": "L1", "Low": "L0"}
SPECIAL_ACTION_BY_RAC = {"High": None, "Serious": "GCS_CONSULT", "Medium": "INCREASE_ASSESSMENT_FREQUENCY", "Low": None}
```

`SPECIAL_ACTION_BY_RAC["High"]`은 여전히 `None`입니다 — High의 special_action은 RAC 단위가 아니라 kill_chain_stage=초기 분기 안에서만 `SPECIAL_ACTION_HIGH_EARLY`로 별도 결정됩니다(진행임박 분기는 special_action 없이 flight_action/payload_action이 이미 충분히 급함).

## 3. payload_action — 리스트형, threat_event 개별 오버라이드 + 기체 스펙 조건부

`WEAPON_DROP`/`DATA_WIPE`가 실제로 뭘 의미하는지 근거 없이 만들어졌던 걸 grill-me로 재점검했습니다. 조사 결과 이 UAV(리모아이-006)는 순수 정찰기체로, 실제 폭발물/탄두를 싣지 않습니다(`T3_위험평가_결정론적_vs_AI.md`: "Group1 UAV는 페이로드 제약상 유인기급 장비를 못 답니다"). `WEAPON_DROP`을 폐기하는 대신, `B-1. 지상통제센터 AI 세부`에 이미 있는 `drone_profile.armament` 필드를 조회하는 조건부 동작으로 재설계해 향후 무장형 기체 확장에 대비했습니다.

```python
# DATA_WIPE — 크립토그래픽 소거(암호화 키 즉시 파기, 물리삭제 아님). PHYSICAL(T3/T4)만 대상.
# 적대적 접촉 위험이 있는 위협에서, 기체가 넘어가도 데이터는 못 넘어가게 한다는 취지.
DATA_WIPE_THREATS = {"T3", "T4"}

# WEAPON_DROP — drone_profile.armament에 expendable 항목이 있을 때만 조건부 실행(무장기체 확장 대비).
# 지금 기체(리모아이-006, EO카메라만)는 armament에 expendable 항목이 없어 사실상 no-op.
# 무게감소로 이탈속도를 높인다는 취지라 총격(T3)에만 적용 — 포획(T4)은 속도로 벗어나는 상황이 아님.
WEAPON_DROP_ELIGIBLE_THREATS = {"T3"}

NAV_MODE_BY_THREAT = {"T1": "INS_ONLY"}


def determine_payload_actions(threat_event: str, drone_profile: dict = None) -> list:
    actions = []
    if threat_event in DATA_WIPE_THREATS:
        actions.append("DATA_WIPE")
    if threat_event in WEAPON_DROP_ELIGIBLE_THREATS:
        armament = (drone_profile or {}).get("armament", [])
        if any(item.get("expendable") for item in armament):
            actions.append("WEAPON_DROP")
    return actions
```

`DATA_WIPE`의 구체 메커니즘: 정찰 데이터(원본 영상·센서로그)는 평소 온보드에 암호화 저장, 키는 휘발성 RAM에만 상주 + GCS와 세션키 동기화. 트리거 시 RAM의 키를 즉시 파기(+ 가능하면 GCS에도 파기 통보). 저장매체 물리삭제(수 분 소요, T4의 몇 초 단위 시간제약과 안 맞음) 대신 키 파기(즉시 완료, 키 없으면 복구 불가) 방식을 택했습니다 — crypto-shredding이라는 표준 보안패턴입니다.

## 4. 행동 결정 함수

```python
def determine_action(threat_event: str, rac: str, kill_chain_stage: str, context: dict = None,
                      drone_profile: dict = None) -> dict:
    """context: {"bearing_deg": float|None, "bearing_source": str|None, "class": str|None}
       (04가 제공하는 threat_event별 원시 위치/분류 정보만. drone_profile은 별도 인자로 분리)
       drone_profile: mission_brief.drone_profile — WEAPON_DROP 조건부 실행용(armament 조회)"""
    category = THREAT_CATEGORY[threat_event]
    context = context or {}
    is_drone_contact = (threat_event == "T4" and context.get("class") == "drone")

    if rac == "High":
        if kill_chain_stage in ("후기", "중기"):
            flight_action = FLIGHT_ACTION_HIGH_EVASIVE[category]
            comms_level = COMMS_LEVEL_HIGH_EVASIVE[category]
            payload_action = determine_payload_actions(threat_event, drone_profile)
            nav_mode = NAV_MODE_BY_THREAT.get(threat_event)
            special_action = None
        else:  # kill_chain_stage == "초기"
            flight_action = FLIGHT_ACTION_HIGH_EARLY
            comms_level = COMMS_LEVEL_HIGH_EARLY
            payload_action = []
            nav_mode = None
            special_action = SPECIAL_ACTION_HIGH_EARLY
    else:
        flight_action = FLIGHT_ACTION_BY_RAC.get(rac, "MAINTAIN")
        comms_level = COMMS_LEVEL_BY_RAC.get(rac, "L0")
        payload_action = []
        nav_mode = None
        special_action = SPECIAL_ACTION_BY_RAC.get(rac)

    return {
        "threat_category": category, "flight_action": flight_action, "comms_level": comms_level,
        "payload_action": payload_action, "nav_mode": nav_mode, "special_action": special_action,
        "is_drone_contact": is_drone_contact,
    }
```

`payload_action`은 `RAC=High`이고 `kill_chain_stage∈{후기,중기}`일 때만 채워지고, 그 외엔 항상 빈 리스트 `[]`입니다(예전엔 `None`이었으나 리스트형으로 통일).

**(신규 확정) context/drone_profile 3출처 분리**: 04↔06/07 사이에 오가는 위치·분류·기체스펙 정보를 하나의 `context` 블롭에 뭉쳐 넣지 않기로 확정했습니다.

| 출처 | 스키마 | 특성 |
|---|---|---|
| `candidates[].context` (04, threat_event별) | `{"bearing_deg": float\|null, "bearing_source": str\|null, "class": str\|null}` | 사이클마다·위협마다 다름. 04가 채널 우선순위 탐색(proximity_object>acoustic_event>rf_spectrum)을 이미 끝내고 단일 값으로 넘김 |
| `cycle_context` (04, 사이클 단위, threat 무관) | `{"optimal_terrain_bearing_deg": float\|null, "lowest_exposure_bearing_deg": float\|null}` | terrain_class 채널 기반, 위협과 무관하게 사이클마다 하나 |
| `mission_brief.drone_profile` (B-1, 미션 단위) | `{"armament": [...]}` | 미션 내내 고정, 04를 거치지 않고 직접 06으로 전달 |

`is_drone_contact`는 T4가 자폭드론 조우(`class="drone"`)인지 표시합니다. `determine_action`/`run_response`가 이제 `drone_profile`을 `context`가 아니라 별도 인자로 받는 것도 이 분리의 직접적인 결과입니다. 실제 04 코드에서 이 스키마로 값을 채워 넘기는 배선은 04/05/06/07 코드 통합 시점으로 보류합니다(13절 한계 참고).

## 5. 전체 파이프라인

```python
def run_response(risk_result: dict, candidate_contexts: dict = None, drone_profile: dict = None) -> dict:
    """candidate_contexts: {threat_event: {"bearing_deg": float|None, "bearing_source": str|None,
       "class": "drone"|"person"|...|None}} — 04가 threat_event별로 넘기는 원시정보(T4 자폭드론
       구분용 class 포함). 07의 bearing 조회도 같은 스키마를 그대로 재사용.
       drone_profile: mission_brief.drone_profile(B-1) — armament 조회용, candidate와 무관한
       미션 단위 고정값이라 candidate_contexts에서 분리해 최상위 인자로 받음."""
    candidate_contexts = candidate_contexts or {}
    candidates = risk_result.get("candidates", [])

    if not candidates:
        ambient_rac = risk_result.get("ambient_rac", "Low")
        return {
            "primary_threat_event": None, "rac": ambient_rac, "kill_chain_stage": None, "threat_category": None,
            "flight_action": FLIGHT_ACTION_BY_RAC.get(ambient_rac, "MAINTAIN"),
            "comms_level": COMMS_LEVEL_BY_RAC.get(ambient_rac, "L0"),
            "payload_action": [], "nav_mode": None,
            "special_action": SPECIAL_ACTION_BY_RAC.get(ambient_rac),
            "secondary_threats": [], "ai_reliability": None,
        }

    primary = candidates[0]
    ctx = candidate_contexts.get(primary["threat_event"], {})
    action = determine_action(primary["threat_event"], primary["rac"], primary["kill_chain_stage"], ctx,
                               drone_profile)

    secondary_threats = [
        {"threat_event": c["threat_event"], "rac": c["rac"],
         "compound_urgency_score": c["compound_urgency_score"], "priority_rank": c["priority_rank"]}
        for c in candidates[1:]
    ]

    return {
        "primary_threat_event": primary["threat_event"], "rac": primary["rac"],
        "kill_chain_stage": primary["kill_chain_stage"], "threat_category": action["threat_category"],
        "flight_action": action["flight_action"], "comms_level": action["comms_level"],
        "payload_action": action["payload_action"], "nav_mode": action["nav_mode"],
        "special_action": action["special_action"], "secondary_threats": secondary_threats,
        "ai_reliability": primary["compound_risk_assessment"]["ai_reliability"],
    }
```

```
category = "PHYSICAL"

```

## 8. 손계산 검증 — 시나리오 3-A: T3 High(RTL) — 무장없는 기체(기본)

새 시나리오: T3(총격)가 RAC=High, kill_chain_stage=후기로 판정(예: LOITER_ROI 정찰중 근접위협 확정, posture 격상 등으로 RAC가 High까지 올라간 경우). `context={}`, `drone_profile=None`(현재 리모아이-006 기준).

```
category = "PHYSICAL"
rac="High", kill_chain_stage="후기" ∈ {후기,중기} → 진행임박 분기
flight_action = "RTL", comms_level = "L3"
payload_action = determine_payload_actions("T3", None)
  "T3" in DATA_WIPE_THREATS → actions=["DATA_WIPE"]
  "T3" in WEAPON_DROP_ELIGIBLE_THREATS → armament=(None or {}).get("armament",[])=[] → any(...)=False → WEAPON_DROP 안 붙음
  → ["DATA_WIPE"]
```

**최종**: `{payload_action:["DATA_WIPE"]}` — 무장이 없으니 WEAPON_DROP은 붙지 않습니다.

## 9. 손계산 검증 — 시나리오 3-B: T3 High(RTL) — 무장기체 가정(확장성 검증, drone_profile 분리 인자)

같은 T3/High/후기 상황이지만, **이제 `drone_profile`은 `context`가 아니라 `run_response()`/`determine_action()`의 별도 인자**로 넘깁니다. `drone_profile={"armament": [{"type":"expendable_pod","expendable": True}]}`로 무장형 기체를 가정합니다(현재 기체 스펙이 아니라 확장성 검증용 가상 시나리오).

```
determine_action("T3", "High", "후기", context={}, drone_profile={"armament":[{"type":"expendable_pod","expendable":True}]})
payload_action = determine_payload_actions("T3", {"armament":[{"type":"expendable_pod","expendable":True}]})
  actions=["DATA_WIPE"]
  armament에 expendable=True 항목 존재 → any(...)=True → actions.append("WEAPON_DROP")
  → ["DATA_WIPE", "WEAPON_DROP"]
```

**최종**: `{payload_action:["DATA_WIPE", "WEAPON_DROP"]}` — 06/07 로직을 전혀 안 바꾸고 `drone_profile`만 다르게 줬는데 자동으로 WEAPON_DROP이 붙습니다. "무장기체 확장 대비" 구조가 실제로 작동함을 확인했습니다.

## 10. 손계산 검증 — 시나리오 4: T7 High — payload_action 없음(NAVIGATION 제외)

```
category = "NAVIGATION"
rac="High", kill_chain_stage="후기" → 진행임박 분기
flight_action = "ALTITUDE_CHANGE_REROUTE", comms_level = "L2"
payload_action = determine_payload_actions("T7", None)
  "T7" not in DATA_WIPE_THREATS({"T3","T4"}) → DATA_WIPE 안 붙음
  "T7" not in WEAPON_DROP_ELIGIBLE_THREATS → WEAPON_DROP 안 붙음
  → []
```

**최종**: `{payload_action:[]}` — T7은 적대적 접촉 상황이 아니라는 설계 원칙대로 아무 payload_action도 안 붙습니다.

## 11. 손계산 검증 — 시나리오 5(신규): T3 High+초기 — POSTURE_ELEVATE, special_action 확정

새 시나리오: T3가 RAC=High이지만 아직 kill_chain_stage=초기(진행임박 아님). 이전엔 `special_action=None`인 미정 상태였으나, 이번 라운드에서 `INCREASE_ASSESSMENT_FREQUENCY`로 확정했습니다(05 재평가 주기 단축 — High는 이미 Serious보다 심각한데 재평가를 Medium 수준으로만 하면 킬체인 진행을 늦게 잡아낼 수 있음).

```
category = "PHYSICAL"
rac="High" → kill_chain_stage="초기" → 진행임박 아님 분기
flight_action = FLIGHT_ACTION_HIGH_EARLY = "POSTURE_ELEVATE"
comms_level = COMMS_LEVEL_HIGH_EARLY = "L1"
payload_action = []
nav_mode = None
special_action = SPECIAL_ACTION_HIGH_EARLY = "INCREASE_ASSESSMENT_FREQUENCY"
```

**최종**: `{flight_action:"POSTURE_ELEVATE", comms_level:"L1", payload_action:[], special_action:"INCREASE_ASSESSMENT_FREQUENCY"}`
