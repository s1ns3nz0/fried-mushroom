
Sensor Abstraction Layer는 `02. UAV Sensor Layer`에서 수집한 원시 센서 데이터를 위협모델링 레이어가 바로 소비할 수 있는 의미 단위(semantic)의 표준 출력으로 바꾸는 계층입니다.

## 설계 원칙 — 왜 이렇게 나누는가

센서 개수를 늘리는 대신, 이미 있는 센서에서 여러 채널을 뽑아내는 게 핵심 원칙입니다. 예를 들어 카메라 한 프레임을 지형 세그멘테이션(terrain_class)·객체탐지(proximity_object) 두 채널에 동시에 재사용합니다. Group1 SWaP(크기·무게·전력) 예산상 센서를 추가하는 것보다 하나의 센서를 여러 목적으로 재해석하는 편이 훨씬 저렴합니다.

같은 이유로, 이 레이어는 AI를 꼭 필요한 채널에만 씁니다. 결정론적 계산(임계값 비교, GIS 조회, 잔차 계산)만으로 충분한 채널은 AI를 쓰지 않고, 규칙만으로는 판별이 안 되는 채널(객체탐지·무기판별, 세그멘테이션 보조, 애매한 음향 구분)에만 AI를 투입합니다. AI를 더 붙이면 좋아질 수 있는 채널은 맨 아래 TO-DO에 따로 정리했습니다.

## 02 → 03 매핑 — 원시 카테고리가 어느 채널로 모이는가

`02. UAV Sensor Layer`의 8개 원시데이터 카테고리는 이 레이어에서 11개 의미론적 채널로 재구성됩니다. 하나의 원시 카테고리가 여러 채널에 나눠 들어가기도 하고, 여러 카테고리가 한 채널로 합쳐지기도 합니다.

| 02번 원시데이터 카테고리 | 관련 원시 항목 | → 03번 채널 |
|---|---|---|
| 1. 영상 정보(EO/IR) | 가시광선/적외선 영상, 짐벌 각도 | `terrain_class`, `proximity_object` |
| 2. 항법 정보(Navigation) | GPS/GNSS+IMU+고도, 기압계, 지자기, 대기속도, 웨이포인트 텔레메트리 | `position_consistency`, `mission_phase` |
| 3. 통신 정보(C2 Link) | 통신 감도/패킷손실, Latency, 공격 로그, 주파수, 암호화 상태 | `link_status`, `link_integrity`, `encryption_status` |
| 4. 전자전 정보(EW) | GNSS 스푸핑 지표, GPS 위성 수신 개수, RF 신호강도 | `position_consistency`(스푸핑 판정 보조지표), `rf_spectrum` |
| 5. 드론 상태(Health) | 배터리, 온도, 진동, 모터 RPM, 낙하산/페일세이프 | `operational_margin`(모터 RPM·진동은 `diagnostics`로, 낙하산/페일세이프는 `failsafe_state`로 노출) |
| 6. 음향 정보(Acoustic) | 총기/폭발음, 적 드론 소리 | `acoustic_event` |
| 7. 환경 정보(Environment) | 풍속/풍향, 기온, 고도(지면기준), 지형고도/경사(DEM) | `operational_margin`(weather_state — 기온 포함), `terrain_class`(GIS 기본값) |
| 8. 임무 상태(Mission Status) | 웨이포인트 진행상태, 목표/귀환 거리, 임무단계 | `mission_phase`, `operational_margin`(time_margin_state). 목표/귀환 거리는 이 레이어에 없음(아래 참고) |
| *(02에 명시 안 됨)* | 레인지파인더/스테레오 거리값 | `obstacle_proximity` |

`obstacle_proximity` 채널은 레인지파인더·스테레오 거리값을 입력으로 쓰는데, 이 원시 항목이 02번 문서의 8개 카테고리에는 명시적으로 없습니다 — 하드웨어 확정 시 02번에 보완이 필요합니다.

목표지점·귀환기지까지 남은 거리는 현재 위치와 항로정보(mission_brief의 corridor·bases)를 합쳐야 계산되는 파생값이라, 이 레이어(원시 센서를 의미단위로 바꾸는 역할)의 범위를 벗어납니다. Threat Modeling Layer(04)나 위험평가 단계에서 다룰 예정입니다.

## 채널별 처리 파이프라인 (11개)

### position_consistency — 🔵 결정론
- **입력**: GPS(lat/lon/alt, HDOP/VDOP), IMU(가속도/각속도), 기압계
- **처리**: GPS 위치와 IMU 관성항법 추정치의 잔차를 계산해(뺄셈) 임계값과 비교
- **출력**: `gps_imu_residual_m`, `baro_residual_m`, `airspeed_residual_ms`, `threshold_m`, `hdop`, `vdop`, `satellite_count`, `cn0_avg_db`

### link_status — 🔵 결정론
- **입력**: C2 라디오 RSSI, SNR/노이즈플로어
- **처리**: 라디오 모뎀이 이미 계산한 값을 그대로 판독
- **출력**: `rssi_dbm`, `noise_floor_dbm`, `freq_mhz`, `packet_loss_rate`, `latency_ms`

### rf_spectrum — 🔵 결정론
- **입력**: 광대역 RF 수신기 스펙트럼 스캔
- **처리**: 임계값 이상탐지
- **출력**: `wideband_anomaly`, `bearing_deg`

### link_integrity — 🔵 결정론
- **입력**: C2 링크 프로토콜 시퀀스번호, 체크섬/서명
- **처리**: 체크섬 실패율 = 실패건수/전체건수
- **출력**: `checksum_fail_rate`, `seq_gap_count`

### encryption_status — 🔵 결정론
- **입력**: C2 링크 프로토콜 암호화 모드 필드
- **처리**: 프로토콜 필드 그대로 판독
- **출력**: `mode`, `downgrade_detected`

### mission_phase — 🔵 결정론적 룰테이블
- **입력**: MISSION_CURRENT, 비행모드, 헤딩/지상속도, 짐벌 매니저 하위상태
- **처리**: declared 값을 짐벌상태·행동패턴과 대조해 mission_phase_confidence 산출
- **출력**: `declared`, `behavioral`, `match`, `mission_phase_confidence`
- **범위 밖**: 목표/귀환 거리는 여기서 계산하지 않음(위 매핑 표 참고)

### terrain_class — 🔵 GIS 우선(1차) + 🟡 카메라 세그멘테이션(보조·검증, 실사용)
- **입력**: 사전 지형DB(GIS), 카메라 raw 프레임
- **처리**: 기본은 GIS 조회. 카메라 결과가 GIS와 다를 때만 세그멘테이션 모델(Fast-SCNN/BiSeNetV2/PIDNet-S)을 가동해 최근 지형변화(벌목 등)를 보정
- **출력**: `dominant_class`, `source`, `gis_last_updated`, `camera_mismatch`, `exposure_score`, `risk_map_ref`

### proximity_object — 🟡 AI(필수)
- **입력**: 카메라 raw 프레임(+선택적 레인지파인더/스테레오)
- **처리**: YOLO 계열(YOLOv8n/YOLO-NAS nano) 객체탐지 → 사람 bbox 크롭 → MobileNetV3-small 무기판별 이진분류. 사람·무기 형태 판별은 규칙기반으로 불가능해 AI가 필수인 채널입니다.
- **출력**: `class`, `weapon_shape`, `bearing_deg`, `closing`, `closure_rate_mps`, `degraded_reason`

### acoustic_event — 🔵 1차 임계값매칭 + 🟡 YAMNet(애매한 경우만 2차, 실사용)
- **입력**: 마이크 raw 파형
- **처리**: 피크dB·rise-time·대역폭으로 1차 필터링해 대부분 확정. 폭죽·근접 프로펠러음처럼 구별이 애매한 경우에만 YAMNet을 2차로 호출
- **출력**: `event_type`, `detection_stage`, `peak_db`, `bearing_deg`

### obstacle_proximity — 🔵 결정론(레인지파인더) 또는 🟡 AI(단안 깊이추정 사용 시)
- **입력**: 레인지파인더/스테레오 거리값
- **처리**: 직접 판독, 또는 레인지파인더가 없으면 단안 깊이추정 모델 사용
- **출력**: `distance_m`, `closure_rate_mps`

### operational_margin (집계, 위협채널 아님) — 🔵 결정론적 worst-case 집계
- **입력**: 배터리(전압/전류/잔량%/온도), 모터/ESC(RPM/온도), IMU 진동, 홈포인트거리, 풍속추정, 시정/기상/기온, 임무경과시간, 낙하산/페일세이프 상태
- **처리**: 배터리/기상(기온 포함)/기계/통신마진/시간마진 5개 항목 중 가장 나쁜 상태를 `overall`로 채택. 낙하산/페일세이프 상태는 이산적 안전상태라 이 worst-case 집계에는 넣지 않고 `failsafe_state`로 별도 노출. 모터 RPM·진동·기온 등 원값은 `diagnostics`에 진단용으로 노출(판단 트리거 아님)
- **출력**: `battery_pct`, `battery_state`, `weather_state`, `mechanical_state`, `link_margin_state`, `time_margin_state`, `overall`, `worst_factor`, `failsafe_state`, `diagnostics`

## 처리 빈도 — 모든 채널을 같은 주기로 돌리지 않는다

11개 채널을 매 프레임 동시처리하는 건 SWaP 예산상 비현실적이라, 채널 성격에 따라 주기를 차등화합니다.

| 구동 방식 | 해당 채널 | 이유 |
|---|---|---|
| 상시·고빈도 | position_consistency, link_status, link_integrity, encryption_status, operational_margin | 연산량이 가볍고(단순 판독/뺄셈) 실시간성이 중요 |
| 상시·저비용 트리거 | acoustic_event | 가벼운 모델이라 상시 가동, 이상 감지 시에만 다음 단계 활성화 |
| 트리거 기반(게이팅) | proximity_object | acoustic_event나 rf_spectrum이 이상신호를 내면 그때 고비용 비전모델 가동 |
| 저빈도(주기적) | terrain_class | 지형은 급변하지 않으므로 초당 1회 이하로도 충분 |

## quality / state / quality_delta — 모든 채널 공통

- **quality**(0.0~1.0): 센서 자체 신뢰도 또는 AI 모델의 예측 확신도. 저시정 상황에서는 카메라 기반 채널(proximity_object, terrain_class)의 quality가 자동으로 낮아지고, 이 값이 이후 위협모델링의 채널가중치 계산에 그대로 반영됩니다.
- **state**(normal/degraded/anomaly): `normal`(정상범위), `degraded`(센서는 작동하지만 신뢰도 낮음 — 예: 저시정), `anomaly`(값 자체가 위협 신호).
- **quality_delta**(신규, 모든 채널 공통 파생값): 전 사이클 대비 quality 변화량(`quality(t) - quality(t-1)`). 새 센서가 아니라 이미 계산하는 quality의 시간차만 추가로 계산하면 됩니다. 안개처럼 서서히 저하되는 상황과, 레이저 조사처럼 순간적으로 급락하는 상황을 구분하는 데 씁니다 — Threat Modeling Layer(04)가 이 값을 T5(레이저) 판정에 사용합니다(proximity_object 또는 terrain_class의 quality_delta가 -0.3 미만으로 급락하면 매칭).

## 출력 포맷

`schema_version`/`id`/`ts`는 사이클당 한 번만 나오는 공통 envelope이고, 채널별 결과는 `channels` 배열 안에 channel/state/quality/**quality_delta**/payload를 담습니다. `id`는 `{sortie_id}-{순번}` 형태로, 사이클마다 순번이 1씩 늘어나 상대적 순서를 표현합니다.

```json
{
  "schema_version": "1.0",
  "id": "GIREOGI-0704-01-4471",
  "ts": 1730620801200,
  "channels": [
    { "channel": "<채널명>", "state": "normal | degraded | anomaly", "quality": 0.0, "quality_delta": 0.0, "payload": { "...채널별 필드..." } }
  ]
}
```

전체 출력 예시, 채널별 payload 필드 상세 타입·값은 `A-1. 추상 결과 세부 내용`에 정리되어 있어 이 문서에서는 중복 생략합니다.

## 통신열화 시 처리 — 이 레이어의 구조화 출력이 곧 대안이다

통신 링크가 열화되면(link_status의 packet_loss_rate·latency_ms 악화, operational_margin의 link_margin_state가 critical), 영상을 그대로 지상국에 보내는 대신 텍스트로 축약해서 보내야 할 수 있습니다. 이때 이미지캡셔닝(VLM)으로 영상을 실시간 설명하는 방식은 채택하지 않았습니다 — 이미 이 레이어가 매 사이클 만들어내는 구조화 JSON(예: `proximity_object`의 `class`, `bearing_deg`, `closure_rate_mps`)을 템플릿으로 텍스트화하는 편이 좌표 정확도가 보장되고, 지연시간이 사실상 0이며, VLM 추론(5초 이상)이 필요 없기 때문입니다.
