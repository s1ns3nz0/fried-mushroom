# F-1. Flight Planning Spec

`07. Flight Planning`의 기술 세부 문서입니다. 실행 가능한 코드, 손계산 검증, 파라미터 출처를 담습니다. 개념·근거·쉬운 설명은 `07. Flight Planning`을 참고하고, 이 문서는 "정확히 어떻게 계산되는가"에 집중합니다.

입력은 `06. Response`의 `run_response()` 출력(`flight_action`, `threat_category`) + `context`(threat_event별, 04가 pre-resolve한 `{"bearing_deg","bearing_source"}`) + `cycle_context`(사이클 단위, threat 무관 `{"optimal_terrain_bearing_deg","lowest_exposure_bearing_deg"}`).

## 1-1. flight_action → speed_mode (신규, `layer_07_planning/speed.py`)

실제 구현은 `altitude.py`의 `_DELTA_MAP`/`compute_altitude_delta` 패턴을 그대로 따릅니다 — flight_action 하나로 값이 정해지는 순수 룩업(추가 판단·이력 없음).

```python
from ..shared.constants import SPEED_MODE_ORDER, SPEED_WEIGHT_DOMINANCE_MARGIN

_MODE_MAP: dict[str, str] = {
    "RTL":                      "MAX",
    "REROUTE":                  "MAX",
    "ALTITUDE_CHANGE_REROUTE":  "MAX",
    "POSTURE_ELEVATE":          "CAUTIOUS",
    "ALTITUDE_CHANGE":          "NORMAL",
    "MAINTAIN":                 "NORMAL",
}

_WEIGHT_ADJUSTABLE_ACTIONS = frozenset({"MAINTAIN", "ALTITUDE_CHANGE", "POSTURE_ELEVATE"})
_SPEED_BY_ORDER = {v: k for k, v in SPEED_MODE_ORDER.items()}  # CAUTIOUS=1<NORMAL=2<MAX=3


def compute_speed_mode(flight_action: str, weights: dict | None = None) -> str:
    """(flight_action, weights) → speed_mode(NORMAL/CAUTIOUS/MAX).

    weights=None 이거나 flight_action이 RTL/REROUTE/ALTITUDE_CHANGE_REROUTE(이미 MAX,
    회피 진행중)면 순수 룩업 그대로. survival-stealth 우세가 SPEED_WEIGHT_DOMINANCE_MARGIN
    (0.1)을 넘으면 한 단계 상/하향(clamp).
    """
    base = _MODE_MAP[flight_action]
    if weights is None or flight_action not in _WEIGHT_ADJUSTABLE_ACTIONS:
        return base

    dominance = weights.get("survival", 0.0) - weights.get("stealth", 0.0)
    order = SPEED_MODE_ORDER[base]
    if dominance > SPEED_WEIGHT_DOMINANCE_MARGIN:
        order = min(order + 1, max(SPEED_MODE_ORDER.values()))
    elif dominance < -SPEED_WEIGHT_DOMINANCE_MARGIN:
        order = max(order - 1, min(SPEED_MODE_ORDER.values()))
    return _SPEED_BY_ORDER[order]
```

`run.py`의 CFIT override(TTC<3s → `ALTITUDE_CHANGE` 강제)가 적용된 뒤의 `effective_action`으로 계산되므로, override가 걸리면 `speed_mode`도 override 이후 값을 따릅니다 — 다만 **CFIT가 걸린 사이클엔 `weights`를 넘기지 않아**(`weights = None if cfit_triggered else cycle_context.get("weights")`) 항상 `NORMAL`로 고정됩니다(SCC-1, 안전 최우선이 운용자 선호보다 우선).

`weights`는 오케스트레이터(`src/onboard/run.py`)가 `cycle_context_07["weights"] = mission_brief.get("weights", {})`로 실어 보냅니다 — corridor_waypoints/bases와 같은 방식(07 `run()` 시그니처 자체는 안 건드림).

## 1-2. MAINTAIN의 reroute_anchor (신규 확정, 이전 결정 번복)

직전 라운드에서 "`replan_scope=NONE`이면 `reroute_anchor=null`"을 계약으로 명시했으나, 이번 라운드에서 뒤집었습니다. `MAINTAIN`일 때 `reroute_anchor="mission_corridor_resume"`을 항상(조건 없이) 부여합니다 — D4D가 좌표를 계산하는 게 아니라, 오토파일럿 자체의 미션시퀀서(현재 활성 웨이포인트 인덱스)에게 통제권을 돌려주는 릴리스 신호입니다. 개념·근거는 `07. Flight Planning` 문서 참고. 조건 분기 없이 상수로 고정하는 이유: MAVLink 미션모드 복귀 신호는 멱등이라 "직전에 회피 중이었는지" 여부를 몰라도 매 사이클 동일하게 보내도 안전하고, 이 파이프라인의 무상태 원칙(ADR-004)과도 맞습니다.

## 1-3. RAC 완화 디바운스 (신규, `layer_07_planning/debounce.py`, ADR-004 명시적 예외)

06은 무상태로 RAC를 매 사이클 재계산하므로(`flight_comms.resolve`), 노이즈로 RAC가 흔들리면 `RTL↔MAINTAIN`이 매 사이클 진동할 수 있습니다. 05가 이미 쓰는 `RAC_ORDER`(High=1<Serious=2<Medium=3<Low=4)를 재사용해 **비대칭 디바운스**를 적용합니다: 악화(숫자 감소)는 즉시 반영, 완화(숫자 증가)만 `FLIGHT_ACTION_DEESCALATE_DEBOUNCE_CYCLES`(3) 사이클 연속일 때 반영.

**(버그 수정, grill-me 3라운드)** 최초 구현은 RAC_ORDER만 비교했으나, `flight_comms.resolve()`가 RAC=High 구간에서 `kill_chain_stage`·`threat_category`에도 좌우된다는 걸 재확인했습니다(High+초기=`POSTURE_ELEVATE`, High+후기+PHYSICAL=`RTL`, 둘 다 RAC_ORDER=1). RAC_ORDER만 보면 RAC=High 유지 중 kill_chain_stage가 진행돼도 "변화 없음"으로 오판해 `POSTURE_ELEVATE`를 고정하는 버그가 있었습니다. `primary_threat_event` 변경(상태 리셋) + `KILL_CHAIN_STAGE_ORDER` 진행(악화 취급) 두 조건을 추가해 고쳤습니다.

```python
# layer_07_planning/debounce.py
from ..shared.constants import (
    FLIGHT_ACTION_DEESCALATE_DEBOUNCE_CYCLES,
    KILL_CHAIN_STAGE_ORDER,
    RAC_ORDER,
)

def apply_debounce(
    rac: str,
    flight_action: str,
    primary_threat_event: str | None,
    kill_chain_stage: str | None,
    state: dict | None,
) -> tuple[str, dict]:
    """(현재 rac/flight_action/primary_threat_event/kill_chain_stage, 이전 debounce_state)
    → (effective_action, new_state).

    state 없음(첫 사이클) 또는 primary_threat_event 변경(새 위협) → 즉시 위임, committed 초기화.
    악화(RAC_ORDER 감소, 또는 동일 RAC에서 kill_chain_stage 진행) → 즉시 반영, streak 리셋.
    완화(RAC_ORDER 증가) → candidate_streak 누적, N도달 시에만 committed 갱신.
    동일 → 변화 없음(committed 유지, streak 리셋).
    """
    current_order = RAC_ORDER[rac]

    if state is None or primary_threat_event != state.get("committed_primary_threat_event"):
        return flight_action, _committed(current_order, flight_action, primary_threat_event, kill_chain_stage)

    committed_order = state["committed_rac_order"]
    stage_progressed = _stage_order(kill_chain_stage) > _stage_order(state.get("committed_kill_chain_stage"))

    if current_order < committed_order or (current_order == committed_order and stage_progressed):
        return flight_action, _committed(current_order, flight_action, primary_threat_event, kill_chain_stage)

    if current_order > committed_order:  # 완화 — 디바운스
        if state.get("candidate_rac_order") == current_order:
            streak = state["candidate_streak"] + 1
        else:
            streak = 1
        if streak >= FLIGHT_ACTION_DEESCALATE_DEBOUNCE_CYCLES:
            return flight_action, _committed(current_order, flight_action, primary_threat_event, kill_chain_stage)
        return state["committed_flight_action"], {
            **state, "candidate_rac_order": current_order, "candidate_streak": streak,
        }

    # 동일 RAC, stage 진행 없음 — 유지
    return state["committed_flight_action"], {**state, "candidate_rac_order": None, "candidate_streak": 0}


def _stage_order(kill_chain_stage):
    return 0 if kill_chain_stage is None else KILL_CHAIN_STAGE_ORDER[kill_chain_stage]


def _committed(rac_order, flight_action, primary_threat_event, kill_chain_stage):
    return {
        "committed_rac_order": rac_order,
        "committed_flight_action": flight_action,
        "committed_primary_threat_event": primary_threat_event,
        "committed_kill_chain_stage": kill_chain_stage,
        "candidate_rac_order": None,
        "candidate_streak": 0,
    }
```

`run.py`에서의 적용 순서(CFIT가 항상 최종 우선):

```python
effective_action, new_debounce_state = debounce.apply_debounce(
    response["rac"], response["flight_action"],
    response["primary_threat_event"], response["kill_chain_stage"],
    debounce_state,
)
# ... target_bearing/altitude_delta는 여기서부터 effective_action 기준으로 계산 ...
# CFIT override는 디바운스 결과보다 항상 우선 (안전 최우선, RAC/디바운스 무관)
if ttc is not None and ttc < TIME_TO_COLLISION_THRESHOLD_S and delta == 0:
    effective_action = "ALTITUDE_CHANGE"
    ...
```

`FlightPlanOutput` 스키마에는 `debounce_state`가 포함되지 않습니다 — 사이클 간 상태는 03의 `previous_qualities`/`extract_qualities` 선례와 동일하게 별도 채널로 흐릅니다: 오케스트레이터가 `extract_flight_plan_state(result)`로 다음 사이클용 상태를 뽑아 `run_cycle(..., previous_flight_plan_state=...)`로 다음 호출에 넘깁니다.

## 2. 방향 계산 헬퍼

```python
def get_bearing(context: dict):
    """context={"bearing_deg":..., "bearing_source":...} — 04가 채널 우선순위 탐색을 이미
       끝내고 넘긴 단일 값(구버전처럼 07이 여기서 flat key를 직접 탐색하지 않음). 없으면 (None, None)."""
    context = context or {}
    bearing = context.get("bearing_deg")
    if bearing is not None:
        return bearing, context.get("bearing_source")
    return None, None

def reverse_bearing(bearing_deg: float) -> float:
    return (bearing_deg + 180) % 360

def compute_local_avoidance(context: dict, cycle_context: dict = None):
    bearing, source = get_bearing(context)
    if bearing is not None:
        return reverse_bearing(bearing), f"threat_reverse({source})"
    fallback_bearing = (cycle_context or {}).get("lowest_exposure_bearing_deg")
    if fallback_bearing is not None:
        return fallback_bearing, "terrain_fallback"
    return None, "undetermined"

def compute_full_reroute(threat_category: str, context: dict, cycle_context: dict = None):
    context = context or {}
    if threat_category in ("PHYSICAL", "REMOTE"):
        bearing, source = get_bearing(context)
        if bearing is not None:
            return reverse_bearing(bearing), f"threat_reverse({source})"
        return None, "last_known_good_position"
    if threat_category == "NAVIGATION":
        optimal_bearing = (cycle_context or {}).get("optimal_terrain_bearing_deg")
        return optimal_bearing, "optimal_terrain"
    return None, "undefined"
```

**(신규 확정) context/cycle_context 분리**: 구버전은 `get_bearing()`이 `context`에서 `f"{ch}_bearing_deg"` 형태의 flat key를 채널 우선순위대로 직접 탐색했고, `optimal_terrain_bearing_deg`/`lowest_exposure_bearing_deg`도 같은 `context` 딕셔너리 안에서 찾았습니다. 이번 라운드에서 04↔06/07 스키마를 재점검하며, threat_event별로 매 사이클 달라지는 값(`bearing_deg`)과 threat와 무관하게 사이클 단위로 하나만 있는 값(지형정보)을 하나의 딕셔너리에 섞는 게 부자연스럽다는 걸 확인해 `context`/`cycle_context` 두 인자로 분리했습니다. `get_bearing()`도 채널 탐색 없이 이미 04가 우선순위를 계산해 넘긴 `context.bearing_deg`/`context.bearing_source`를 그대로 읽도록 단순화했습니다.

## 3. 지시값 계산

```python
def compute_directive(flight_action: str, threat_category: str, context: dict = None,
                       cycle_context: dict = None) -> dict:
    context = context or {}
    replan_scope = REPLAN_SCOPE_BY_FLIGHT_ACTION.get(flight_action, "NONE")

    if flight_action == "MAINTAIN":
        return {"target_bearing_deg": None, "altitude_delta_m": 0,
                "replan_scope": "NONE", "reroute_anchor": None}

    if flight_action == "POSTURE_ELEVATE":
        return {"target_bearing_deg": None, "altitude_delta_m": POSTURE_ELEVATE_ALTITUDE_M,
                "replan_scope": "LOCAL", "reroute_anchor": "altitude_only"}

    if flight_action == "ALTITUDE_CHANGE":
        return {"target_bearing_deg": None, "altitude_delta_m": ALTITUDE_DELTA_PREVENTIVE_M,
                "replan_scope": "LOCAL", "reroute_anchor": "altitude_only"}

    if flight_action == "RTL":
        bearing, anchor = compute_local_avoidance(context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": 0,
                "replan_scope": "LOCAL", "reroute_anchor": anchor}

    if flight_action == "REROUTE":
        bearing, anchor = compute_full_reroute(threat_category, context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": 0,
                "replan_scope": "FULL", "reroute_anchor": anchor}

    if flight_action == "ALTITUDE_CHANGE_REROUTE":
        bearing, anchor = compute_full_reroute(threat_category, context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": ALTITUDE_DELTA_TERRAIN_M,
                "replan_scope": "FULL", "reroute_anchor": anchor}

    return {"target_bearing_deg": None, "altitude_delta_m": 0, "replan_scope": "NONE", "reroute_anchor": None}
```

`POSTURE_ELEVATE` 분기는 `ALTITUDE_CHANGE`와 완전히 같은 패턴(고도만 조정, `reroute_anchor="altitude_only"`)이고 상승폭만 `POSTURE_ELEVATE_ALTITUDE_M`(25m)으로 다릅니다. `MAINTAIN`을 명시적으로 먼저 분기 처리해 `replan_scope="NONE"`인 다른 케이스(구버전엔 `POSTURE_ELEVATE`도 여기 속했음)와 혼동되지 않게 했습니다.

## 4. 전체 파이프라인

```python
def run_flight_planning(response_result: dict, context: dict = None, cycle_context: dict = None) -> dict:
    """06(response_v2.run_response)의 출력을 받아 기동 지시값을 계산.
       context: threat_event별 pre-resolved {"bearing_deg","bearing_source"}
       cycle_context: 사이클 단위 {"optimal_terrain_bearing_deg","lowest_exposure_bearing_deg"}"""
    flight_action = response_result.get("flight_action", "MAINTAIN")
    threat_category = response_result.get("threat_category")
    directive = compute_directive(flight_action, threat_category, context, cycle_context)
    return {"flight_action": flight_action, **directive}
```

## 8. 손계산 검증 — 시나리오 4: T7 High(ALTITUDE_CHANGE_REROUTE, cycle_context 분리)

## 9. 손계산 검증 — 시나리오 5(신규): POSTURE_ELEVATE(+25m)

새 시나리오: `06`에서 RAC=High+kill_chain_stage=초기로 `flight_action="POSTURE_ELEVATE"`가 나온 경우. context/cycle_context 둘 다 불필요(고도만 조정).

```
flight_action=="POSTURE_ELEVATE" 분기 → altitude_delta_m=POSTURE_ELEVATE_ALTITUDE_M=25, target_bearing_deg=None, reroute_anchor="altitude_only"
```

**최종**: `{target_bearing_deg:null, altitude_delta_m:25, replan_scope:"LOCAL", reroute_anchor:"altitude_only"}` — `ALTITUDE_CHANGE`(15m)와 같은 패턴이지만 값이 더 큽니다.

## 10. 손계산 검증 — 시나리오 6(신규): T4 High(RTL, bearing 없음 → terrain_fallback)

새 시나리오: RTL인데 `context`에 bearing_deg가 없고, `cycle_context`에 `lowest_exposure_bearing_deg=150`만 있는 경우(방향탐지 실패, 지형 폴백만 가능).

```
flight_action=="RTL" 분기 → compute_local_avoidance({}, {"lowest_exposure_bearing_deg":150})
  get_bearing({}): bearing_deg 없음 → (None, None)
  fallback_bearing = cycle_context.get("lowest_exposure_bearing_deg") = 150 → (150, "terrain_fallback")
altitude_delta_m = 0
```

**최종**: `{target_bearing_deg:150, altitude_delta_m:0, replan_scope:"LOCAL", reroute_anchor:"terrain_fallback"}` — `lowest_exposure_bearing_deg`가 이제 `cycle_context`에서 조회됨을 확인.

## 12. 수평 회피 궤적 — bearing offset (`route.py` 확장, 신규)

REROUTE / ALTITUDE_CHANGE_REROUTE(replan_scope=FULL) 시 `target_bearing_deg`(위협 반대방향)를 받아 모든 corridor waypoint를 해당 방향으로 `ROUTE_EVASION_OFFSET_M`(= 100m) 평행이동합니다.

### 12-1. 적용 조건

```
if replan_scope == "FULL" and target_bearing_deg is not None:
    waypoints = _apply_bearing_offset(waypoints, target_bearing_deg)
```

RTL은 `flight_action == "RTL"` 조건에서 미리 분기되어 offset 적용 없이 반환됩니다.
ALTITUDE_CHANGE / POSTURE_ELEVATE는 LOCAL scope이므로 조건을 만족하지 않습니다.

### 12-2. 좌표 계산

```python
import math

EARTH_RADIUS_M = 6_371_000.0    # constants.py
ROUTE_EVASION_OFFSET_M = 100.0  # constants.py

def _apply_bearing_offset(waypoints, bearing_deg):
    bearing_rad = math.radians(bearing_deg)
    d = ROUTE_EVASION_OFFSET_M
    result = []
    for wp in waypoints:
        lat = float(wp["lat"])
        lon = float(wp["lon"])
        d_lat = math.degrees(d * math.cos(bearing_rad) / EARTH_RADIUS_M)
        d_lon = math.degrees(d * math.sin(bearing_rad) / (EARTH_RADIUS_M * math.cos(math.radians(lat))))
        result.append({**wp, "lat": lat + d_lat, "lon": lon + d_lon})
    return result
```

위도·경도만 이동하고 `alt_m`/`clearance_m`은 보존합니다. offset 후 `_apply_delta()`가 고도를 처리합니다.

### 12-3. 손계산 검증 — bearing=0°(북향), lat=37.5°, d=100m

```
d_lat = degrees(100 * cos(0) / 6_371_000) = degrees(1.5694e-5) ≈ +0.000899°
d_lon = degrees(100 * sin(0) / ...) = 0
```

→ 위도 +0.000899° ≈ 북쪽 100m 이동. 경도 불변.
