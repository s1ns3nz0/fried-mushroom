# F-1. Flight Planning Spec

`07. Flight Planning`의 기술 세부 문서입니다. 실행 가능한 코드, 손계산 검증, 파라미터 출처를 담습니다. 개념·근거·쉬운 설명은 `07. Flight Planning`을 참고하고, 이 문서는 "정확히 어떻게 계산되는가"에 집중합니다.

입력은 `06. Response`의 `run_response()` 출력(`flight_action`, `threat_category`) + `context`(threat_event별, 04가 pre-resolve한 `{"bearing_deg","bearing_source"}`) + `cycle_context`(사이클 단위, threat 무관 `{"optimal_terrain_bearing_deg","lowest_exposure_bearing_deg"}`).

---

## 1. flight_action → 재계획 범위

```python
REPLAN_SCOPE_BY_FLIGHT_ACTION = {
    "RTL": "LOCAL",
    "REROUTE": "FULL",
    "ALTITUDE_CHANGE_REROUTE": "FULL",
    "ALTITUDE_CHANGE": "LOCAL",
    "MAINTAIN": "NONE",
    "POSTURE_ELEVATE": "LOCAL",  # 예방적 고도상승은 실제로 고도를 바꾸므로 LOCAL(altitude_only와 동일 패턴)
}

ALTITUDE_DELTA_PREVENTIVE_M = 15
ALTITUDE_DELTA_TERRAIN_M = 50
# RAC=High인데 kill_chain_stage=초기(아직 진행임박 아님)일 때의 예방적 고도상승폭.
# Serious(+15)보다 크게 잡음 — High는 Serious보다 이미 더 심각한 등급이라는 걸 고도값에도 반영.
POSTURE_ELEVATE_ALTITUDE_M = 25
```

## 2. 방향 계산 헬퍼

```python
def get_bearing(context: dict):
    """context={"bearing_deg":..., "bearing_source":...} — 04가 채널 우선순위 탐색을 이미
       끝내고 넘긴 단일 값(구버전처럼 07이 여기서 flat key를 직접 탐색하지 않음). 없으면 (None, None)."""
    context = context or {}
    bearing = context.get("bearing_deg")
    if bearing is not None:
        return bearing, context.get("bearing_source")
    return None, None

def reverse_bearing(bearing_deg: float) -> float:
    return (bearing_deg + 180) % 360

def compute_local_avoidance(context: dict, cycle_context: dict = None):
    bearing, source = get_bearing(context)
    if bearing is not None:
        return reverse_bearing(bearing), f"threat_reverse({source})"
    fallback_bearing = (cycle_context or {}).get("lowest_exposure_bearing_deg")
    if fallback_bearing is not None:
        return fallback_bearing, "terrain_fallback"
    return None, "undetermined"

def compute_full_reroute(threat_category: str, context: dict, cycle_context: dict = None):
    context = context or {}
    if threat_category in ("PHYSICAL", "REMOTE"):
        bearing, source = get_bearing(context)
        if bearing is not None:
            return reverse_bearing(bearing), f"threat_reverse({source})"
        return None, "last_known_good_position"
    if threat_category == "NAVIGATION":
        optimal_bearing = (cycle_context or {}).get("optimal_terrain_bearing_deg")
        return optimal_bearing, "optimal_terrain"
    return None, "undefined"
```

**(신규 확정) context/cycle_context 분리**: 구버전은 `get_bearing()`이 `context`에서 `f"{ch}_bearing_deg"` 형태의 flat key를 채널 우선순위대로 직접 탐색했고, `optimal_terrain_bearing_deg`/`lowest_exposure_bearing_deg`도 같은 `context` 딕셔너리 안에서 찾았습니다. 이번 라운드에서 04↔06/07 스키마를 재점검하며, threat_event별로 매 사이클 달라지는 값(`bearing_deg`)과 threat와 무관하게 사이클 단위로 하나만 있는 값(지형정보)을 하나의 딕셔너리에 섞는 게 부자연스럽다는 걸 확인해 `context`/`cycle_context` 두 인자로 분리했습니다. `get_bearing()`도 채널 탐색 없이 이미 04가 우선순위를 계산해 넘긴 `context.bearing_deg`/`context.bearing_source`를 그대로 읽도록 단순화했습니다.

## 3. 지시값 계산

```python
def compute_directive(flight_action: str, threat_category: str, context: dict = None,
                       cycle_context: dict = None) -> dict:
    context = context or {}
    replan_scope = REPLAN_SCOPE_BY_FLIGHT_ACTION.get(flight_action, "NONE")

    if flight_action == "MAINTAIN":
        return {"target_bearing_deg": None, "altitude_delta_m": 0,
                "replan_scope": "NONE", "reroute_anchor": None}

    if flight_action == "POSTURE_ELEVATE":
        return {"target_bearing_deg": None, "altitude_delta_m": POSTURE_ELEVATE_ALTITUDE_M,
                "replan_scope": "LOCAL", "reroute_anchor": "altitude_only"}

    if flight_action == "ALTITUDE_CHANGE":
        return {"target_bearing_deg": None, "altitude_delta_m": ALTITUDE_DELTA_PREVENTIVE_M,
                "replan_scope": "LOCAL", "reroute_anchor": "altitude_only"}

    if flight_action == "RTL":
        bearing, anchor = compute_local_avoidance(context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": 0,
                "replan_scope": "LOCAL", "reroute_anchor": anchor}

    if flight_action == "REROUTE":
        bearing, anchor = compute_full_reroute(threat_category, context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": 0,
                "replan_scope": "FULL", "reroute_anchor": anchor}

    if flight_action == "ALTITUDE_CHANGE_REROUTE":
        bearing, anchor = compute_full_reroute(threat_category, context, cycle_context)
        return {"target_bearing_deg": bearing, "altitude_delta_m": ALTITUDE_DELTA_TERRAIN_M,
                "replan_scope": "FULL", "reroute_anchor": anchor}

    return {"target_bearing_deg": None, "altitude_delta_m": 0, "replan_scope": "NONE", "reroute_anchor": None}
```

`POSTURE_ELEVATE` 분기는 `ALTITUDE_CHANGE`와 완전히 같은 패턴(고도만 조정, `reroute_anchor="altitude_only"`)이고 상승폭만 `POSTURE_ELEVATE_ALTITUDE_M`(25m)으로 다릅니다. `MAINTAIN`을 명시적으로 먼저 분기 처리해 `replan_scope="NONE"`인 다른 케이스(구버전엔 `POSTURE_ELEVATE`도 여기 속했음)와 혼동되지 않게 했습니다.

## 4. 전체 파이프라인

```python
def run_flight_planning(response_result: dict, context: dict = None, cycle_context: dict = None) -> dict:
    """06(response_v2.run_response)의 출력을 받아 기동 지시값을 계산.
       context: threat_event별 pre-resolved {"bearing_deg","bearing_source"}
       cycle_context: 사이클 단위 {"optimal_terrain_bearing_deg","lowest_exposure_bearing_deg"}"""
    flight_action = response_result.get("flight_action", "MAINTAIN")
    threat_category = response_result.get("threat_category")
    directive = compute_directive(flight_action, threat_category, context, cycle_context)
    return {"flight_action": flight_action, **directive}
```

## 5. 손계산 검증 — 시나리오 1: T3 Serious(ALTITUDE_CHANGE)

`E-1` 6절 이어받음: `flight_action="ALTITUDE_CHANGE"`, `threat_category="PHYSICAL"`, context/cycle_context 없음.

```
flight_action=="ALTITUDE_CHANGE" 분기 → altitude_delta_m=15, target_bearing_deg=None, reroute_anchor="altitude_only"
```

**최종**: `{target_bearing_deg:null, altitude_delta_m:15, replan_scope:"LOCAL", reroute_anchor:"altitude_only"}`

---

## 6. 손계산 검증 — 시나리오 2: T4 High(RTL, bearing_deg=45)

`E-1` 7절 이어받음: `flight_action="RTL"`, `threat_category="PHYSICAL"`, `context={"bearing_deg": 45, "bearing_source": "proximity_object"}`(04가 카메라 채널을 우선순위 1위로 이미 골라서 넘김).

```
flight_action=="RTL" 분기 → compute_local_avoidance(context, None)
  get_bearing(context): context.bearing_deg=45 존재 → (45, "proximity_object")
  reverse_bearing(45) = (45+180) mod 360 = 225
  → (225, "threat_reverse(proximity_object)")
altitude_delta_m = 0 (RTL 분기는 고도 안 건드림)
```

**최종**: `{target_bearing_deg:225, altitude_delta_m:0, replan_scope:"LOCAL", reroute_anchor:"threat_reverse(proximity_object)"}`

---

## 7. 손계산 검증 — 시나리오 3: T1 High(REROUTE, bearing 없음)

새 시나리오: T1(GPS스푸핑) 위협이 RAC=High, kill_chain_stage=후기로 판정됐다고 가정(REMOTE 카테고리이므로 `06`에서 `flight_action="REROUTE"`, `comms_level="L2"` 산출). 방향탐지 실패로 `context={}`(bearing_deg 없음).

```
flight_action=="REROUTE" 분기 → compute_full_reroute("REMOTE", {}, None)
  threat_category="REMOTE" → get_bearing({}): bearing_deg 없음 → (None, None)
  bearing is None → return (None, "last_known_good_position")
altitude_delta_m = 0
```

**최종**: `{target_bearing_deg:null, altitude_delta_m:0, replan_scope:"FULL", reroute_anchor:"last_known_good_position"}`

방향 정보 없이도 "마지막 정상 상태로 복귀"라는 명확한 재계획 기준이 나옵니다 — bearing이 없다고 재계획 자체가 불가능해지지 않습니다.

---

## 8. 손계산 검증 — 시나리오 4: T7 High(ALTITUDE_CHANGE_REROUTE, cycle_context 분리)

새 시나리오: T7(지형충돌) 위협이 RAC=High, kill_chain_stage=후기로 판정(NAVIGATION 카테고리, `06`에서 `flight_action="ALTITUDE_CHANGE_REROUTE"`). terrain_class 채널이 사전계산한 `optimal_terrain_bearing_deg=300`(가장 트인 지형 방향)이 **`cycle_context`**(threat_event별 `context`가 아님)에 실려있다고 가정.

```
flight_action=="ALTITUDE_CHANGE_REROUTE" 분기 → compute_full_reroute("NAVIGATION", {}, {"optimal_terrain_bearing_deg":300})
  threat_category="NAVIGATION" → optimal_bearing = cycle_context.get("optimal_terrain_bearing_deg") = 300
  → (300, "optimal_terrain")
altitude_delta_m = ALTITUDE_DELTA_TERRAIN_M = 50
```

**최종**: `{target_bearing_deg:300, altitude_delta_m:50, replan_scope:"FULL", reroute_anchor:"optimal_terrain"}`

고도상승(50m)이 방향 재계획과 함께 나가서 "먼저 높이 뜨고, 동시에 트인 방향으로 향한다"는 설계 의도(07 문서의 "고도상승이 1차 방어선")가 그대로 반영됩니다.

---

## 9. 손계산 검증 — 시나리오 5(신규): POSTURE_ELEVATE(+25m)

새 시나리오: `06`에서 RAC=High+kill_chain_stage=초기로 `flight_action="POSTURE_ELEVATE"`가 나온 경우. context/cycle_context 둘 다 불필요(고도만 조정).

```
flight_action=="POSTURE_ELEVATE" 분기 → altitude_delta_m=POSTURE_ELEVATE_ALTITUDE_M=25, target_bearing_deg=None, reroute_anchor="altitude_only"
```

**최종**: `{target_bearing_deg:null, altitude_delta_m:25, replan_scope:"LOCAL", reroute_anchor:"altitude_only"}` — `ALTITUDE_CHANGE`(15m)와 같은 패턴이지만 값이 더 큽니다.

## 10. 손계산 검증 — 시나리오 6(신규): T4 High(RTL, bearing 없음 → terrain_fallback)

새 시나리오: RTL인데 `context`에 bearing_deg가 없고, `cycle_context`에 `lowest_exposure_bearing_deg=150`만 있는 경우(방향탐지 실패, 지형 폴백만 가능).

```
flight_action=="RTL" 분기 → compute_local_avoidance({}, {"lowest_exposure_bearing_deg":150})
  get_bearing({}): bearing_deg 없음 → (None, None)
  fallback_bearing = cycle_context.get("lowest_exposure_bearing_deg") = 150 → (150, "terrain_fallback")
altitude_delta_m = 0
```

**최종**: `{target_bearing_deg:150, altitude_delta_m:0, replan_scope:"LOCAL", reroute_anchor:"terrain_fallback"}` — `lowest_exposure_bearing_deg`가 이제 `cycle_context`에서 조회됨을 확인.

## 12. 알려진 한계 · 다음 과제

| 구분 | 내용 |
|---|---|
| 연동 | `context`(threat_event별 bearing_deg/bearing_source)와 `cycle_context`(optimal_terrain_bearing_deg/lowest_exposure_bearing_deg) 분리 스키마는 이번 라운드에서 확정됐지만, 04/03에서 실제로 이 스키마대로 값을 채워 07까지 넘기는 배선은 아직 없음 — 특히 `lowest_exposure_bearing_deg`/`optimal_terrain_bearing_deg`는 terrain_class의 exposure_score를 방위각별로 미리 계산해두는 새로운 전처리 로직이 필요(03 확장 사안, 이번 라운드 범위 밖) |
| 설계 | REMOTE(T1)의 방향탐지 가능 여부(rf_spectrum이 지향안테나를 갖췄는지)가 하드웨어에 달려있어, 실제로 bearing이 있는 경우가 얼마나 될지 검증 안 됨 |
| 데이터 | ALTITUDE_DELTA_PREVENTIVE_M(15m)/POSTURE_ELEVATE_ALTITUDE_M(25m)/ALTITUDE_DELTA_TERRAIN_M(50m) 모두 실측 이전 팀 설정값 — 실제 UAV 상승률·장애물 평균 고도분포로 재조정 필요 |
| 경계 | 07 출력(target_bearing_deg 등) 이후의 실제 궤적최적화(Risk-A*/MPC)는 D4D 범위 밖으로 명시적으로 뒀는데, 오토파일럿 쪽 인터페이스 스펙(MAVLink 메시지 타입 매핑)은 이번 라운드에서도 범위 밖으로 명시적으로 제외(이후 별도 라운드) |
| 코드 | 04/05/06/07을 실제 `d4d_pipeline/`에 통합하는 작업은 여전히 별도 라운드로 보류 |
