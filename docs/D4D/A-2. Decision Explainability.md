# A-2. Decision Explainability — 온보드 결정 근거 설명

## 왜 (임팩트)

국방 자율 시스템에서 **왜 그 결정을 내렸는가**는 운용자 신뢰·감사(auditability)·사후검토의
핵심이다. D4D 파이프라인은 03→07 각 단계에 풍부한 provenance(`confidence_source`,
`kill_chain_stage`, RAC 유도, `reroute_anchor` 등)를 이미 실어 나르지만, 이를 사람이 읽는
**결정 근거로 엮어주는 백엔드가 없었다** — 대시보드의 "AI 결정 설명" 워크스루(#269)도
프런트 목업 단계일 뿐 실데이터 근거가 없었다.

`onboard/explain.py`는 그 백엔드다: `run_cycle` 결과 하나를 받아 **관측→추론→결정→행동**의
단계별 근거를 구조화해 산출한다.

## 무엇 (동작)

`explain_cycle(result) -> dict`:

| step | 근거 |
|---|---|
| 03 센서 추상화 | 비정상 채널 목록 (state≠normal) |
| 04 위협 판정 | 지배 위협 T-코드·설명·확신도 + provenance(confidence_source/kill_chain_stage/outcome) |
| 05 위험 평가 | RAC + L등급/심각도/urgency (후보 없으면 ambient_rac 근거) |
| 06 대응 결정 | flight_action + comms/nav/payload/threat_category |
| 07 비행 계획 | replan_scope·bearing·고도Δ·speed·reroute_anchor·waypoint 수 |

`format_explanation(explanation) -> str`: 로그·CLI·debrief 표시용 텍스트 렌더.

### 예시 (T3 골든)

```
결정: ALTITUDE_CHANGE — T3(근접 소화기) 위협, RAC=Serious
  [03] 센서 추상화: 11개 채널 중 비정상 2개: proximity_object, acoustic_event
  [04] 위협 판정: T3 (근접 소화기) 확신도 0.917
        provenance: confidence_source=..., kill_chain_stage=...
  [05] 위험 평가: RAC=Serious (L=B, S=Critical)
  [06] 대응 결정: 비행행동=ALTITUDE_CHANGE
  [07] 비행 계획: replan=LOCAL, 방위=128.42°, 고도Δ=15m, 속도=NORMAL
```

## CRITICAL — 파생 읽기전용

- 파이프라인 출력을 **읽어 조립만** 한다. 결정을 절대 바꾸지 않고 입력 결과 dict 를
  변이하지 않는다(`derived_readonly=True`, 무변이 회귀 테스트로 강제).
- 결정론 로직·상수를 읽거나 쓰지 않으며 `shared.THREAT_CATALOG`(설명 조회, 읽기전용)만 참조.
- 부분/빈 결과에도 예외 없이 best-effort 동작 (모든 골든 시나리오 설명 가능 회귀).

## 용도 / 소비처

- 대시보드 "AI 결정 설명" 워크스루(#269)의 **실데이터 소스**.
- 사이클 로그/debrief 의 사람이 읽는 근거.
- 감사 추적(어느 신호→어느 위협→어느 RAC→어느 행동).

테스트: `tests/integration/test_explain.py` (9건 — 구조·provenance·무위협분기·무변이·전
골든 설명가능·텍스트 렌더).
