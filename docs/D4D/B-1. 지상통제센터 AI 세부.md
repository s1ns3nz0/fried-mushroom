# B-1. 지상통제센터 AI 세부

`01. 지상 정보 센터 AI`(한글 서술)의 기술 상세본입니다. 팀이 확정한 METT+TC 상태 모델(단일 출처 스키마)을 그대로 옮기고, 이 중 지상통제센터 AI가 채우는 부분과 온보드 레이어가 채우는 부분을 구분해 표시했습니다. 필드명·구조 자체는 원본 스키마를 그대로 따릅니다 — 이 스키마가 정식 기준이며, 임의로 이름을 바꾸지 않습니다. `civil_sensitivity_estimate`처럼 원본 스키마에 없는 필드는 팀 확장분으로 별도 표시했습니다.

## 1. 최상위 구조

상태 모델은 두 갈래로 나뉩니다.

- **`drone_profile`** — METT+TC 밖 최상위 필드. 기체 고정 특성(항속·탑재 무장·센서 스위트 등)이라 임무마다 바뀌는 동적 상황과 분리합니다.
- **`mettc`** — 임무 동적 상태. 6요소 **M / E / T_terrain / T_troops / T_time / C**로 구성됩니다.

`drone_profile`이 밖인 이유: 기체 스펙은 임무 시작 시 1회 정해진 뒤 비행 중 변하지 않는 상수인 반면, METT+TC는 관측·시간에 따라 매 tick 갱신되는 변수입니다 — 상수와 변수를 섞지 않습니다.

## 2. 지상통제센터 AI가 채우는 필드 vs 온보드가 채우는 필드

**출처** 약어: `set_mission`=GCS 운용자 입력(WS `/ws`) · `world_init`=board↔world 지형 blob 복원 · `findings:rf/vlm/internal`=추상화 semantic JSON의 source별 finding · `obs`=self_state passthrough 실측값 · `내부시계`=tick·경로 진행 계산 · `c4i_verify`=C4I 자동데이터 대조(팀 확장, 등록값 검증용).
**갱신주기** 약어: `1회`=임무 시작 1회 · `tick`=매 tick · `재평가`=재평가 트리거 시.

`set_mission`(+ 일부 `world_init`) 소스는 지상통제센터 AI(이 문서)가 임무 시작 전 채우는 필드입니다. `findings:*` / `obs` / `내부시계`는 비행 중 온보드 레이어(`02. UAV Sensor Layer`, `03. Sensor Abstraction Layer`)가 채우는 필드로, 이 문서에서는 참고용으로만 표시합니다.

### drone_profile (최상위) — 전부 지상통제센터 AI(운용자 등록)

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `id` | 기체 식별자 | string | set_mission | 1회 |
| `type` | 기체 종류(quadrotor 등) | string | set_mission | 1회 |
| `endurance_rated_s` | 정격 체공 시간(초) | int | set_mission | 1회 |
| `spare_available` | 예비 드론 보유 여부 | bool | set_mission (+ c4i_verify) | 1회 |
| `armament` | 정적 탑재 무장 목록 `[{type, expendable}]` | array | set_mission | 1회 |
| `sensor_suite` | 탑재 센서 종류 목록 | string[] | set_mission | 1회 |

- `spare_available`은 운용자가 직접 등록하지만, 등록 즉시 C4I 자산관리체계 자동 데이터와 대조합니다. 불일치 시 확신도를 조정하는 게 아니라 "등록값이 C4I 기록과 다릅니다" 경고만 표시합니다(사실 확인이지 확률 문제가 아니므로). 이 대조는 팀이 추가한 확장 로직이며 원본 스키마에는 없습니다.

### M (Mission) — 지상통제센터 AI(운용자 등록)

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `posture` | 태세 `{defcon, infocon, watchcon}` | object | set_mission | 1회 / 태세 변경 시 |
| `higher_intent` | 상급 부대 임무·의도 | string | set_mission | 1회 |
| `unit_mission` | 우리 부대 임무 | string | set_mission | 1회 |
| `uav_mission` | UAV 임무 `{name, purpose, type, goal}` | object | set_mission(+`goal`은 world_init) | 1회 |
| `weights` | 임무 가치 가중치 `{stealth, survival, info_quality, timeliness}` 각 축 절대값 ∈[0,1] | object | set_mission | 1회 / 운용자 재설정 시 |

- `posture`: 워치콘(감시)·인포콘(정보전)·데프콘(방어준비) 태세. 숫자가 작을수록 높은 경계 — 태세가 높을수록 위험 평가가 보수화(위협 임계 하향·가중치 강화)합니다.
- `weights`: 사람이 GCS에서 설정하는 절대 가중치(0~1). 위험 평가의 `mission_loss` 산출 근거이자 "판단 가중치의 출처가 사람"임을 고정하는 값입니다.

### T_terrain (Terrain & weather) — corridor만 지상통제센터 AI, 나머지는 온보드

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `terrain_ref` | 온보드 지형 `np.ndarray[H, W]` in-process 참조 | ref | world_init | 1회 *(온보드)* |
| `H`, `W` | 지형 격자 크기 | int | world_init | 1회 *(온보드)* |
| `hmin`, `hmax` | 표고 최소·최대(m) | float | world_init | 1회 *(온보드)* |
| `corridor` | 회랑 `{type, axis, half_width, alt_min, alt_max}` | object | set_mission | 1회 |
| `weather` | 기상(확장 필드) | object\|null | findings:vlm | 재평가 *(온보드)* |

- `corridor`: 이동 가능 구역 — `axis`(중심축 polyline) + `half_width`(반폭) + `alt_min`/`alt_max`(고도 범위)로 정의. 회랑 이탈은 위험 평가의 하드 제약으로, 위반 시 penalize·판단 트리거됩니다. `01. 지상 정보 센터 AI`의 "GCS 항로정보"가 바로 이 필드입니다.

### T_troops.friendly — bases/assets는 지상통제센터 AI, 자기상태는 온보드

Troops는 "자기상태 + 가용 아군 전투력"입니다. `pos`/`battery`/`gps_quality`/`comms_q`/`sensors_ok`/`armament_state`는 비행 중 온보드가 채우는 자기상태이고, `friendly`(우군 기지·자산)는 지상통제센터 AI가 임무 시작 시 채우는 부분입니다.

`friendly.bases[i]` — 복귀(rtb) 목적지 후보:

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `id` | 기지 식별자(rtb `target.base_id`가 참조) | string | set_mission + world_init | 1회 |
| `pos` | 기지 위치 [x, y] | float[2] | set_mission + world_init | 1회 |
| `type` | 기지 종류 `home`(원기지)·`forward`(전진기지)·`emergency`(비상착륙점) | string | set_mission + world_init | 1회 |
| `available` | 사용 가능 여부(GCS/브리핑 설정) | bool | set_mission + world_init | 1회 |
| `reachable` | 현 위치·`T_time.endurance_s`로 도달 가능 여부 | bool | 파생(위험 평가) | tick *(온보드)* |

`friendly.assets[i]` — 우군 자산(다른 UAV·부대), 상황 인식용(확장 슬롯, 판단 소비는 미정):

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `id` | 자산 식별자 | string | set_mission + world_init | 1회 |
| `kind` | 자산 종류(uav·unit 등) | string | set_mission + world_init | 1회 |
| `pos` | 추정 위치 [x, y] | float[2] | set_mission + world_init | 1회 |
| `status` | 자산 상태(active 등) | string | set_mission + world_init | 1회 |

`bases`·`assets`는 GCS 임무 브리핑(`set_mission` 확장) + world_init로 설정되는 정적 정보입니다. `reachable`은 파생 필드 — 위험 평가가 매 tick 현 위치·`T_time.endurance_s`로 도달 가능성을 판정합니다(온보드 담당).

### C (Civil considerations) — no_fly_zones는 지상통제센터 AI, civil_sensitivity_estimate는 팀 확장

| 필드 | 의미 | 타입 | 출처 | 갱신주기 |
|---|---|---|---|---|
| `no_fly_zones` | 비행 금지 구역 목록 | array | set_mission | 1회 |
| `civil_areas` | 민간 지역 `[{id, center, radius, roe_relevant}]` | array | findings:vlm | 재평가 *(온보드)* |
| `civil_sensitivity_estimate` ⚠️팀 확장 | 작전지역 전체의 민간 밀집도 추정(단일값) | enum(`low`/`medium`/`high`) | 하이브리드(C4I 초안 + 운용자 확정) | 1회 / 승인 시 |

- `no_fly_zones`는 `01. 지상 정보 센터 AI`의 "GCS 항로정보"가 참조하는 금지구역 DB와 같은 개념입니다.
- `civil_sensitivity_estimate`는 원본 스키마에 없는 필드입니다. 기존 위험평가 로직이 심각도 계산 입력으로 이 값을 쓰고 있어서, 스키마 확장 규칙("스키마에 없으면 추가 가능")에 따라 보존했습니다. `civil_areas`(온보드, 지역별, 실시간)와는 별개로 임무 전체를 대표하는 단일값만 유지하기로 했습니다 — 지금 위험평가가 임무 단위로 계산되기 때문입니다. 나중에 온보드 `civil_areas`와 실제 연동이 필요해지면 지역별로 확장을 재검토합니다.

## 3. 참고 — 온보드가 채우는 나머지 필드 (요약)

아래는 비행 중 온보드 레이어가 채우는 필드로, 지상통제센터 AI의 책임 범위 밖입니다. 상세는 `02. UAV Sensor Layer`, `03. Sensor Abstraction Layer`, `A-1. 추상 결과 세부 내용`을 참고하세요.

| 요소 | 필드 | 비고 |
|---|---|---|
| E (Enemy) | `tracks[]` | 상황 추적기가 `track_id` 기준으로 갱신. A→B 이동은 동일 트랙의 위치 변화이지 신규 개체가 아닙니다 |
| T_troops | `pos`, `battery`, `gps_quality`, `comms_q`, `sensors_ok`, `armament_state` | self_state passthrough. `armament`(정적 탑재)와 `armament_state`(현재 잔량·투기 가능여부)는 구분됨 |
| T_time | `elapsed_s`, `eta_goal_s`, `endurance_s` | 내부시계·경로진행·배터리소비율 모델로 계산 |
| T_terrain | `weather` | findings:vlm |
| C | `civil_areas` | findings:vlm |

## 4. 지상통제센터 AI의 기존 3소스 모델과의 관계

`01. 지상 정보 센터 AI`에서 설명한 GCS·운용자·C4I 3소스는 이 스키마에서 전부 `set_mission`(GCS 운용자 입력, WS `/ws`) 한 창구로 들어옵니다. 즉 데이터의 논리적 출처(GCS 자체 계산, 운용자 직접입력, C4I 자동피드)는 여전히 셋으로 구분되지만, 시스템에 실제로 반영되는 기술적 진입점은 GCS의 `set_mission` API 하나입니다 — 운용자가 임무를 등록하는 한 번의 행위 안에 세 소스의 정보가 모두 합쳐져 들어갑니다.

기존 문서 대비 정리된 변경점(팀 논의로 확정):

- `spare_asset_available`(구, C4I 자동 피드로 설명) → `drone_profile.spare_available`(운용자가 기체 프로필에 직접 등록)로 소속 변경. 다만 등록값을 C4I 자산관리체계 데이터와 대조(`c4i_verify`, 팀 확장)해 불일치 시 경고하는 검증 로직을 추가해 정확성을 보완합니다.
- 단일 `rtb_point` → `T_troops.friendly.bases[]`(원기지/전진기지/비상착륙점 여러 후보)로 확장
- `corridor` 개념은 동일 유지, 필드 구조만 `{type, axis, half_width, alt_min, alt_max}`로 공식화
- `weights`(임무 가치 가중치: 은닉성/생존성/정보가치/적시성)는 신규 추가 — 운용자가 직접 설정하는 값
- `civil_sensitivity`(구, C4I 초안+운용자 확정) → `C.civil_sensitivity_estimate`(팀 확장 필드)로 보존. 작전지역 전체 단일값 유지, `civil_areas`(온보드·지역별)와는 별개 필드로 관리

## 5. 지상통제센터 AI 조립 규약 (신규 — 2026-07 라운드)

layer 01 구현(`src/gcs/layer_01_info_center/`)이 비행 전 `{drone_profile, mettc}` 상태모델을
조립할 때 따르는 규약. 온보드 파이프라인(02-07)은 이 상태모델의 **투영본**(6필드
MissionBrief)을 소비하며, 투영 매핑과 초기값 규약을 여기 고정한다.

### 5.1 온보드-소유 필드 초기값 (비행 전 시점)

§2 의 `obs`/`내부시계` 소스 필드는 지상 조립 시점에 관측이 없으므로 초기값을 넣는다:

| 필드 | 초기값 | 근거 |
|---|---|---|
| `T_time.elapsed_s` | `0` | 임무 시작 전 |
| `T_time.eta_goal_s` | `null` | 미계산 (온보드 몫) |
| `T_time.endurance_s` | `drone_profile.endurance_rated_s` (없으면 `null`) | 정격값에서 시작 |
| `T_troops.pos` | home base pos (없으면 corridor axis 첫 점), alt `null` | 이륙 전 위치 |
| `T_troops.battery` | `drone_profile.battery_pct / 100` (없으면 `null`) | 등록값 |
| `T_troops.gps_quality / comms_q / sensors_ok` | `null` | 미관측 |
| `T_terrain.terrain_ref` | `"onboard_dem"` (stub) · H/W/hmin/hmax `null` | 실 DEM 후순위 |
| `T_terrain.weather` | `null` | findings:vlm (온보드) |
| `E.tracks[].history` | `[]` · `last_seen_tick` `null` | 지상 시점 무 tick |
| `C.civil_sensitivity_estimate` | `civil_density_draft` 최고 density (없으면 `"low"`) | C4I 초안 요약 |

### 5.2 C4I 입력 정규화

C4I 피드는 `enemy_tracks[]`(구조화)를 정본으로 하되, 레거시 문자열
`enemy_situation[]` 이 오면 `{kind:"report", label:<문자열>, confidence:0.5}` 트랙으로
승격 수용한다. `E.tracks` 에는 `source:"c4i"` 를 태깅해 출처를 보존한다.
